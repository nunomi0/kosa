## 객체지향 프로그래밍(OOP; Objected Oriented Programming)

- 여러 개의 작은 설계도를 모아 큰 설계도를 만드는 행위
    - ex) 결제, 배송, 재고, 회원, 상품, 판매처, 카드 → 업무별 별도의 설계도
    - 설계도 1장으로 모든 업무를 표현 불가능
        - 현업 기업 환경에서 한 개의 클래스만 설계해서 개발 불가
        - 설계도 여러 장으로 나누어서 개발 (기준, 논리적)
    
    → 설계도끼리 관계 설정은 어떻게 할까?
    
- 클래스와 클래스 사이의 관계를 맺는 방법
    1. 상속: is~a / 은(는) ~이다
        - JAVA: child extends Base
    2. 포함: has ~ a / 은(는) ~ 가지고 있다 (부품 정보로 다른 객체를 가지고 있는 것)
- 특징
    1. 다중 상속 불가
        - Java에서는 한 클래스가 부모 클래스를 오직 하나만 가질 수 있다
    2. 계층적 상속
        - 상속을 계층적으로 쭉 이어갈 수 있다
    3. 인터페이스는 예외적으로 다중 구현 가능

### 상속

1. 장점
    1. 재사용성
        - 이미 만들어둔 코드를 다시 쓰기 위함
2. 단점
    1. 초기 설계 비용 큼
        - 부모 클래스는 여러 자식이 물려받을 공통 설계도라서 한 번 정하면 바꾸기 어려움
    2. 커플링 (Coupling, 결합도)
        - 상속받는 순간 자식은 부모에게 강하게 묵임
    3. 디커플링 (Decoupling, 다형성)
        - 커플링을 느슨하게 풀어내는 것
    4. 인터페이스
        - 다형성을 클래스 상속으로만 구현하면 여전히 부모 구현에 묶이므로, 구현을 빼고 규약만 남김

```java
class GrandFather { // extend Object 가 생략되어 있음
	public int gmoney=5000;
	private int pmoney=10000; // 상속 관계에서 접근 불가
	public GrandFather() {
		System.out.println("GrandFather");
}

class Father extends GrandFather{
	public int fmoney=3000;
	public Father() {
		System.out.println("Father");
	}
}
class Child extends Father{
	public int cmoney=1000;
	public Child() {
		System.out.println("Child");
	}
}
```

```java
Child child new Child();
// 메모리에 올라가는 순서
// Object -> GrandFather -> Father -> Child
```

- 원은 점을 가지고 있다

```java
class Point{
	int x;
	int y;
	
	Point(){
		// this.x=1;
		// this.y=2;
		this(1,2);
	}
	Point(int x, int y){

		this.y=y;
	}

```

- 원은 도형이다 상속 (원 extedns 도형)

```java
class Circle extends Shape{
	Point point; // 포함(부품)
	int r; // 구체화, 특수화
	
	Circle {
		// this.point = new Point(10,15);
		// this.r=10;
		this(new Point(10,15), 10);
	}
	
	// overloading 걸어서 점과 반지름을 입력 받기
	Circle(Point point, int r){
		this.point = point;
		this.r = r;
	}
}
```

### 상속 관계에서 override(재정의)

- 상속 관계에서 자식이 부모의 함수를 재정의 (내용을 바꾼다)
- `상속 관계` 에서 `자식 클래스`가 `부모 클래스` 메서드의 `재정의(내용)` 할 수 있다
- ex) toString()
- 문법
    1. 함수 이름 부모와 동일
    2. parameter, return type 동일
    3. 실행 블럭 `{ }` 내부 코드만 바뀜
- `@Override` 어노테이션
    - ‘나 이거 오버라이드 한 거야’ 라고 컴파일러에게 알려 주면 시그니처가 안 맞을 때 에러로 알려줌
        - 시그니처(Signiture)
            - 자바가 메서드를 식별하는 최소 단위
            - 메서드 이름, 파라미터 목록으로 구성(개수, 타입, 순서)

```java
class Point2{
	int x = 4;
	int y = 5;
	
	String getPosition() {
		return this.x + "/" + this.y;
	}
}

class Point3D extends Point2{
	int z = 6;
	
	@Override // 툴과 컴파일러가 읽어서 문법적으로 문제가 있는지 검증
	String getPosition() {
		return this.x + "/" + this.y + "/" + this.z;
}
```

### 동적 바인딩

- 재정의는 엄청나게 강력 → 부모 함수로 가더라도 재정의가 있다면 재정의 함수로 간다
- 접근은 자식의 재정의된 함수만 접근 가능
- `super()` 키워드로 부모 함수에 접근 가능
- 인터페이스에서 사용 가능

---

- 오버로딩
    - 하나의 이름으로 여러가지 기능을 제공하는 함수
    - 하나의 함수가 파라미터의 개수와 타입을 다르게해서 다르게 쓰는 것
    - 개발자가 쉽게 사용할 수 있다
- 오버라이딩
    - 상속 관계에서 재정의
    - 상속 관계에서 부모 함수를 재정의해서 사용하는 방법
- 연관(관계)
    - DB 관계
        - 1:1, 1:N, M:N
        - ex) 사원과 부서와의 관계
        - DB에서 1:N 관계는 객체지향에서 연관관계로 풀어짐

## this

객체가 메모리에 만들어지고, 그 객체가 자기 자신을 부르는 방법

1. 객체 자신을 가리키는
    - `this.필드명`, `this.메서드명()`
2. 생성자를 호출하는 this
    - 생성자가 여러 개 있을 때(오버로딩) 중복된 초기화 코드를 줄임
    - 여러 개의 생성자 호출 가능

## final

- 더 이상 바뀌지 못하게 잠근다
- 위치별 의미
    1. 변수에 final → 값 잠금(상수)
        
        ```java
        final int NUM = 0;   // 재할당 금지
        ```
        
    2. 클래스에 final → 상속 금지
        
        ```java
        public final class Math {
            // ...
        }
        ```
