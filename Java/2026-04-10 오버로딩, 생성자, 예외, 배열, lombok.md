### (복습) static, block 어디에 저장되고, 언제 만들어지고, 언제 사라지는가?

- static
    - static variable: 객체 간 공유 자원
    - static 함수: 객체 없이 호출 가능한 함수 (편하게 쓰려고 만듦)
    - 저장 위치: Method Area
    - 생성 시점: 해당 클래스가 처음 로딩될 때
    - 소멸 시점: 해당 클래스가 언로드될 때 (프로그램 종료 시)
    - static 초기화 이후 인스턴스 변수 초기화
        
        → static 초기화 시 일반 자원이 안 보이는 상태 (static은 초기화 시점에 인스턴스 자원 참조 불가능)
        
        → static 끼리 선언 추천
        
- block
    
    ```java

            int y = 2;  // y는 이 블록 안에서만 유효
    ```

## 객체지향 프로그래밍(OOP)

- 개념: 여러 개의 작은 조각(설계도)를 모아서 큰 조각(큰 설계도)를 만드는 행위
    - AOP(Aspected-Oriendted Programming): 중복되는 부분을 가로로 잘라서 한 곳에 모으는 것
        - ex) 로그, 보안 트랜잭션에서 사용 가능
- 특징
    1. 상속
        - 재사용성
    2. 캡슐화
        - 간접 할당을 통해 데이터나 출력을 조작
        - private, setter, getter
    3. 다형성
        - 하나의 타입(상속 관계에서 부모 타입)이 여러 개의 객체 주소를 가지는 것
            - 상속 관계에서 부모 타입 변수는 자식 객체의 주소를 가질 수 있다

## 오버로딩(Overloading)

### method overloading

- 하나의 이름으로 여러 가지 기능을 하는 함수
- parameter 개수, 타입 다름을 인정한다
    - ex) System.out.println()
- 성능
    - 오버로딩의 사용은 성능과는 무관하다고 생각하자
    - 개발자가 편하게 사용하는데 목적이 있음
    - 컴파일러가 판단해야 하므로 오히려 느릴 수 있다
- 사용하지 않아도 문제가 되지 않는다
    - Java API에는 이미 오버로딩이 적용되어 있다
- 문법
    1. 함수의 이름 동일
    2. parameter 개수와 타입, 순서가 다름
        - ex) m(int i, String s) ≠ m(Stirng s, int i)
    3. return type은 오버로딩의 판단 기준이 아님
    4. parameter의 순서가 다름을 인정

> 오버로딩: 하나의 이름으로 여러 가지 기능을 하는 함수
오버라이딩: 부모의 메서드를 상속 받아 내용을 덮어쓰는 것

## 생성자 함수(constructor)

- 특수한 목적을 가진 함수
    - 생성되는 객체마다 강제적으로 member field 초기화 할 수 있음
        - 자유도가 높은 것은 객체지향에서 부정적 → 강제성 필요
- 일반 함수와 다른 점
    - 함수 이름이 고정 (class 이름과 동일)
    - return type이 없음 (default void)
    - 생성자 overloading을 통해 개발자가 무엇을 강제할 수 있다.
- 여러 개의 생성자 정의 가능 → 중복 코드가 생기므로 this() 사용
    
    ```java
    class User {
        String name;
        int age;
        String email;
        
        User(String name) {
            this(name, 0);  // ← User(String, int) 생성자 호출
        }
        
        User(String name, int age) {
            this(name, age, "");  // ← User(String, int, String) 생성자 호출
        }
        
        User(String name, int age, String email) {
            this.name = name;   // 실제 초기화는 여기서만
            this.age = age;
            this.email = email;
        }
    }
    ```
    
- private 생성자 → 싱글톤의 시작
    - 싱글톤: 객체를 하나만 만들어서 공유하는 것
    
    ```java
    public class Car {
    	private Car() {} // private 선언 시 외부에서 new 불가능
    }
    ```
    

명시적으로 생성자를 만들지 않으면 자동으로 default 생성자 생성됨

- java Car + enter → 컴파일러가 기본 생성자 만듦 (public Car() {})


## this

1. `this.필드`
    - 객체 자신을 가리키는 this
    - 앞으로 생성될 객체의 주소를 담을 곳이라 가정하고 사용
    - 인스턴스 자식을 가리키는 참조 변수, 인스턴스의 주소가 저장될 곳
2. `this()`
    - 생성자를 호출하는 this
    - 그 주소를 가지고 재호출(여러 번) 가능 (원칙적으로는 생성자는 객체 생성 시 한 개만 호출)
    - 단 this 사용하면 예외적으로 여러 개의 생성자를 호출할 수 있다.
    
    ```java
    // this(”red”, “auto”, 2); → 할당하는 중복 코드를 감소
    
    class Zcar{
    	String color;
    	String gearType;
    	int door;
    	
    	Zcar(){
    		this("red","auto",2); // 1. 여기서
    		System.out.println("default"); // 4. 함수 종료까지 진행
    	}
    	
    	Zcar(String color , String gearType , int door){ // 2. 여기로 들어감
    		this.color = color;
    		this.gearType = gearType;
    		this.door = door;
    		System.out.println("overloading"); // 3. overloading 출력
    	}
    }
    ```

## Exception

### 오류

1. 에러(error)
    - 컴퓨터 하드웨어의 고장으로 인해 응용 프로그램 실행 오류가 발생하는 것
        - 네트워크 장애, 메모리 등
    - 개발자가 코드적으로 해결 불가
2. 예외(exception)
    - 잘못된 사용 또는 코딩으로 인한 오류
    - 예외가 발생되면 프로그램은 곧바로 종료
        - 예외 처리를 통해 실행 상태 유지 가능
    - 개발자가 코드 실수 발생시킴 → 문제 발생 원인 파악 → 수정 → 배포
- 컴파일시 안되면 해결 가능
- 런타임시 안되면 코드 실행 안됨
    - 프로그램 실행 도중 발생되는 것
    - 문제 발생 → 강제 종료 → 문제가 생긴 시점 이후 코드는 실행이 안됨

### 예외 처리

- 프로그램 안정적 종료되고 나서 분석(로그, 상태 코드)하는 것
- 분석이 끝나면 코드 수정 및 재배포

```java
try {
	// 문제가 의심되는 코드 넣기
	// 문제가 발생되면 그 문제를 담을 수 있는 객체가 자동으로 만들어짐
	// 객체에 정보가 담겨서 catch로 던져짐
	// ex) [0][1][2] -> [3] = 100 예외 발생시 new ArrayIndexOutofBoundException("...")
} catch (Exception e) { // e라는 변수가 예외 객체의 주소를 받음
	// 문제 파악(결과 출력)
	// 처리 (코드가 자동 수정되지는 않음)
	// 보고 필요 (1. 관리자 이메일로 전송 / 2. 로그 기록(장애 발생 기록))
	// 강제 종료는 방어됨
} finally {
	// 문제가 발생하던, 발생하지 않던 강제적으로 실행되는 코드
	// 함수 종료 (return, kill) -> return을 만나도 finally 강제 실행
	// ex) DB 작업 시 자원 해제(연결 객체 소멸) 수행 (conn.close())	
}
```
